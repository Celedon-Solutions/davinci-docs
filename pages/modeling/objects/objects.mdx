## Base Fields

All Objects in Davinci have the following Fields that can be edited or defined.

| Property        | Description                                                                                     |
| --------------- | ----------------------------------------------------------------------------------------------- |
| `Name`          | The name of the object.                                                                          |
| `Shortname`     | An acronym, ID, or other short name for the object. This is used when displaying the object as a reference in equations or other anchor elements in code. |
| `Documentation` | The written summary, intent, or description of the object.                                        |

In addition to the base fields objects also have a `lock` status which will prevent editing of the object until it is unlocked. 

<Note>Locking (or unlocking) will automatically lock all of children elements and their descendents. This action can be undone by unlocking those child elements.</Note>


## Ways Objects Connect

Most Davinci objects can connect to other objects in several ways. These connections are what make the model useful as a structured engineering graph rather than a flat list of records.

### Parent And Child Hierarchy

Objects live in a tree. Each object has a parent and may own child objects. This hierarchy is visible in the model index and is used to organize systems, packages, requirements, behaviors, artifacts, and reference material.

Parent-child hierarchy is best for ownership and structure. For example, a system part may own subsystem parts, attributes, ports, and requirements.

### References

Reference Objects serve as a truth source or origin of data within the model. These objects are uploaded as files and used by Davinci to source data for the model.

For non-reference objects, references connect model content back to source material that informed the object, value, structure, or claim. References can point to documents, PDFs, source snippets, pages, anchors, or generated evidence depending on the workflow.

Use references when you need traceability to source material or want the Davinci Agent to ground model changes in evidence.

See [Files And References](/pages/editor/files-and-references) for the workflow overview and [Reference Objects](/pages/modeling/objects/reference) for field-level details.

### Relationships

Relationships connect objects across the model without changing the ownership hierarchy. They are used for traceability, allocation, dependencies, interfaces, validation, requirements subjects, mitigations, source links, and other engineering connections. Some relationships are implict given an object's type or purpose.

Relationships can be incoming, outgoing, or bidirectional depending on the relationship type.

See [Relationships](/pages/modeling/common/relationships) for the relationship type reference.

### Tags

Tags provide a flexible way to categorize objects across the project regardless of where they live in the hierarchy. Use tags for ownership, maturity, status, subsystem, priority, requirement compliance, or any other project-specific grouping.

Tags can be applied from the Properties View and managed in project settings.

See [Tags](/pages/editor/project-settings/tags) for tag creation, filtering, restrictions, and deletion behavior.
